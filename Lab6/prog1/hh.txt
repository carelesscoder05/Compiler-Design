### Test Case 1: Pure Constant Folding
**The Concept:** Constant folding is an optimization where the compiler evaluates mathematical expressions containing only constants at compile-time, rather than leaving them for the program to calculate at run-time.

**What is happening:**
1. The lexer reads `10` and `20` as `NUM` tokens. Your parser stores `a` and `b` in the symbol table with their `isConst` flags set to `1`.
2. When the parser hits `c = a + b`, it looks up `a` and `b` in the symbol table. Because both have `isConst == 1`, the `E PLUS E` rule triggers the `if($1.isConst && $3.isConst)` block.
3. Instead of generating a string `"10 + 20"`, it physically does the math in C (`10 + 20 = 30`) and passes `30` up the syntax tree. This optimized constant cascades down through the multiplication and division steps.

**The Output:**
```text
a = 10
b = 20
c = 30
d = 90
e = 45
```

### Test Case 2: Algebraic Identities
**The Concept:** Algebraic simplification removes redundant mathematical operations that don't change the value of a variable, saving CPU cycles.

**What is happening:**
1. `x` is stored as a constant with the value `5`.
2. When evaluating `y = x + 0`, the parser evaluates the `E PLUS E` rule. It notices the right-hand side (`$3`) is a constant and its value is `0`. 
3. It hits your specific optimization block: `else if($3.isConst && $3.value == 0) { $$ = $1; }`. This completely bypasses the addition and simply copies the properties of `x` directly into `y`. The same logic applies to multiplying by `1`.

**The Output:**
```text
x = 5
y = 5
z = 5
w = 5
v = 5
```

### Test Case 3: Mixed Variables and Constants
